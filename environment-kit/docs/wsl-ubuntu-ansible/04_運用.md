# 04 運用

セットアップ・検証・トラブルシュートの実務手順。コマンドはすべて [`platform/wsl-ubuntu/`](../../platform/wsl-ubuntu/) 直下で実行する（ターゲットの正は [`Makefile`](../../platform/wsl-ubuntu/Makefile)）。

```bash
cd /home/ubuntu/repos/private-kit/environment-kit/platform/wsl-ubuntu
```

## 1. 最短導線

最初に迷ったら次だけ実行すればよい。

```bash
make setup-ansible
make syntax
make setup-base
make setup
make check
```

必要に応じて:

```bash
make setup-user
make test-rust
make setup-rust
make help
```

## 2. Windows / Ubuntu 側の初手

Windows 側で人手実施:

```powershell
wsl --update
wsl --version
wsl --set-default-version 2
```

必要に応じて:

```powershell
wsl --shutdown
```

`.wslconfig` の例:

```ini
[wsl2]
memory=16GB
processors=8
swap=8GB
localhostForwarding=true
```

Ubuntu 側で最初に入れる最低限パッケージ:

```bash
sudo apt update
sudo apt upgrade -y
sudo apt install -y \
  build-essential \
  curl \
  wget \
  git \
  unzip \
  zip \
  ca-certificates \
  gnupg \
  lsb-release \
  software-properties-common \
  apt-transport-https \
  jq \
  tree \
  htop \
  net-tools \
  dnsutils \
  iproute2 \
  make \
  pkg-config
```

clone:

```bash
mkdir -p ~/repos
cd ~/repos
git clone <YOUR_REPOSITORY_URL> private-kit
```

GitHub への SSH 公開鍵登録は手動のままにする。  
一方で、登録済み秘密鍵の配布と `git clone` 時の利用は Doppler シークレット（例: `INTERNAL_GIT_SSH_PRIVATE_KEY`）で自動化できる。

## 3. セットアップ導線

共通で次の extra-vars を使える。

```bash
PROFILE=full|minimal|runtime|ci
PLATFORM=wsl|molecule
```

- `PROFILE`
  - 導入範囲の切替。通常運用は `full`、軽量導線は `minimal`、runtime 検証は `runtime`、Molecule は `ci` を使う。
- `PLATFORM`
  - 環境差分の切替。通常運用は `wsl`、Molecule は `molecule` を使う。

### 最小（基盤のみ）

```bash
make setup-ansible   # Ansible / OpenSSH / make の初期導入
make syntax PROFILE=minimal
make setup-base      # WSL 基盤（locale / timezone / wsl.conf / Python・pipx・uv / Doppler）
```

`make setup-base` の実体は `playbooks/converge/base.yml`。`make syntax` は新しい主入口 `playbooks/main.yml` の構文チェックを行う。

### opt-in: ユーザー初期化

```bash
make setup-user      # Git 設定 / ~/.gitignore_global / ~/.bashrc / ~/.ssh 権限
```

Git の `user.name` / `user.email` は推測しない。必要なら Ansible 変数（`git_user_name` / `git_user_email`）で与える。

### 標準（フル）

```bash
make setup           # config.yml 駆動（playbooks/compat/site-selective.yml）
```

`make setup` は既存運用を壊さないため、互換導線の `playbooks/compat/site-selective.yml` を使う。新構造の主入口は `playbooks/main.yml` だが、選択式 UI はまだ `config.yml` 駆動の互換レイヤを正とする。

導入対象（既定）: Rust / Node.js / Go / Docker / Kubernetes ツール群 / gcloud / AWS CLI・Terraform・gh / PostgreSQL。Java・Kotlin は opt-in（既定 OFF）。component の ON/OFF は [`config.yml`](../../platform/wsl-ubuntu/ansible/config.yml) で調整する。

### 個別: Rust だけ

```bash
make test-rust       # playbooks/converge/runtime-rust.yml の構文チェック
make setup-rust      # Rust 単体（selective の状態に依存しない導線）
```

### 並列実行（高速化）

`playbooks/compat/site-parallel.yml` を直接実行する（Makefile ターゲット未定義）。base 完了後に development-runtime / aws / postgresql を並列 async で実行する。

```bash
cd ansible
ANSIBLE_CONFIG=./ansible.cfg ansible-playbook playbooks/compat/site-parallel.yml -i inventory/hosts -e wsl_profile=full -e execution_platform=wsl
```

## 4. 検証

```bash
make test            # syntax + lint + check
make check           # playbooks/check.yml で導入済みツールの状態を確認
make check-system    # 補助スクリプトで導入済みツールのバージョン/状態を確認
make version         # ansible --version
```

- 各 role は `validate_installation`（既定 true）で導入物を検証する。スキップしたい場合は `-e validate_installation=false`。
- 構文だけ見たい場合は各 playbook に `--syntax-check`。`make syntax` は `playbooks/main.yml`、`make test-base` は `playbooks/converge/base.yml`、`make test-rust` は `playbooks/converge/runtime-rust.yml` を見る。
- `make check` は `taskfiles/verify/` を通じて `core` / `runtime` / `container` / `security` / `cloud` / `data` を順に確認する。

## 4.1 Molecule

Docker 一時環境で role / playbook を検証する。初回は仮想環境を作るため少し時間がかかる。

```bash
make molecule-install
make molecule-list
make molecule-test
make molecule-test-base
make molecule-test-user
```

- `molecule-test` は最小 `default` smoke scenario を実行する。
- `default` / `base` / `user-bootstrap` を基準 scenario とし、まずここを green に保つ。
- 個別 scenario を直接叩く場合は `make molecule-test SCENARIO=runtime` のように指定する。
- `user-bootstrap` は role 単体の idempotence 検証に向く。
- `base` は `cache` + `base` の代表シナリオ。
- `runtime` / `postgresql` / `docker` は重いので、通常は変更箇所に応じて scenario を絞る。
- `molecule-test-all` は最後の総合確認に回す。
- Molecule は Docker コンテナで実行するため、`systemd` に依存する service 起動やホストの timezone / locale 変更は `PLATFORM=molecule` 相当の変数で抑制している。そこは `playbooks/converge/base.yml` や `playbooks/main.yml` の実機確認で補完する。

## 5. Doppler 駆動ブートストラップ（クラウド / コンテナ向け）

リポジトリを clone してシークレットを注入しつつ playbook を流す導線。詳細仕様は [`script/bootstrap-from-doppler.md`](../../platform/wsl-ubuntu/script/bootstrap-from-doppler.md)。

```bash
DOPPLER_TOKEN=<service-token> \
REPO_HTTPS_URL=<repo-url> \
ANSIBLE_PLAYBOOK=playbooks/main.yml \
  bash platform/wsl-ubuntu/script/bootstrap-from-doppler.sh
```

- 秘密値は `doppler run -- ansible-playbook` 経由で注入し、リポジトリには置かない。
- SSH クローン時は `INTERNAL_GIT_SSH_PRIVATE_KEY` を優先し、未設定時のみ後方互換で `DEPLOY_KEY_PRIVATE` を使う。
- Doppler のシークレット名/プロジェクトは登録済みの値を正とする（自分から削除・改名しない）。

## 6. ポート方針

| 用途 | ポート |
|---|---|
| WSL apt PostgreSQL | `5432` |
| Docker PostgreSQL dev / test / dwh | `5433` / `5434` / `5435` |

Docker DB は [`docker-database/`](../../platform/wsl-ubuntu/docker-database/) の Makefile で起動する。

## 7. トラブルシュート

- **`make test` が log_path で警告/失敗する**: `ansible.cfg` の `log_path`（`/home/ubuntu/.ansible/ansible.log`）が書込不可だと警告が出る。書込可能なパスを用意するか環境変数 `ANSIBLE_LOG_PATH` で上書き。
- **PATH が通らない（rustc/go/gcloud が見つからない）**: `/etc/profile.d/*.sh` はログインシェルで source される。`source /etc/profile.d/rust.sh` 等で即時反映、または再ログイン。
- **PostgreSQL に接続できない**: ネイティブは `5432`、Docker は `5433-5435`。`make check` でサービス状態と接続を確認。
- **冪等性**: 各タスクは `creates:` / `command -v` でガード済み。再実行は基本安全。
- **Molecule は通るが実機で差が出る**: Docker コンテナでは `systemd` や WSL 固有挙動を再現しない。Molecule は role の回帰検知、実機 playbook は運用確認という役割分担で扱う。
