# 02 移行ロードマップ

構成の進化の経緯と、改善提案の現状（実装済 / 一部 / 未着手）を整理する。本書が改善ロードマップの正であり、旧 `wsl-ubuntu-ansible-improvement-plan.md` の要点もここに吸収した。詳細な背景メモとしては [`ansible/docs/refactoring-plan.md`](../../platform/wsl-ubuntu/ansible/docs/refactoring-plan.md) / [`ansible/docs/improvement-proposals.md`](../../platform/wsl-ubuntu/ansible/docs/improvement-proposals.md) を参照する。なお、`ansible/docs/` は現行仕様の正本ではなく、背景資料・旧設計アーカイブとして扱う。

## 1. これまでの進化

初期は単一 playbook（`playbooks/site.yml`）+ スクリプトのワンショット構成だった。現在は次まで進んでいる。

- **単一 SSOT 化**: バージョン/基盤設定を [`group_vars/all.yml`](../../platform/wsl-ubuntu/ansible/group_vars/all.yml) に集約。
- **role 分割**: `development-runtime` をサブタスク（rust/nodejs/golang/java/kotlin）に分解。`base` / `gcp` / `user-bootstrap` / `cache` を新設。
- **実行導線の多様化**: フル / 基盤のみ / 選択式 / Rust 単体 / ユーザー初期化 / 並列の 6 playbook。
- **検証の組み込み**: 各 role の `validate.yml`、`script/check-system.sh`。
- **言語/クラウド方針の確定**: ML=Python / 非 ML=Rust、メインクラウド=GCP、Java/Kotlin は opt-in。

## 3. 今後の候補（優先度の目安）

- **中**: group_vars の分割（規模が増えた場合）。
- **低〜中**: `monitoring` role。
- **低**: `backup` role。

いずれも実装はコード側の変更を伴うため、本書はステータス追跡に留め、着手時に [03_実装カタログ.md](03_実装カタログ.md) と同期させる。
