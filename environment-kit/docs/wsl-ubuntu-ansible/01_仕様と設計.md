# 01 仕様と設計

WSL Ubuntu 向け開発環境を Ansible で再現可能にするための、目的・スコープ・設計原則をまとめる。実装の正は [`platform/wsl-ubuntu/`](../../platform/wsl-ubuntu/) のコードであり、本書はその設計意図を説明する。現行の変数設計は単一ファイルではなく、[`ansible/group_vars/all.yml`](../../platform/wsl-ubuntu/ansible/group_vars/all.yml) をベースに `profiles` / `platform` / `domains` を重ねる layered 構成を正とする。

## 1. 目的

- WSL 上の Ubuntu に、開発・インフラ・データベースの一式を **冪等に** セットアップする。
- 手順を人手のチェックリストではなく Ansible role + Makefile に落とし、再構築・別マシン展開を容易にする。
- バージョン、feature flag、platform 差分、個人設定の責務を分け、ドキュメントと実体の乖離を防ぐ。

## 2. スコープ（対象）

- WSL2 上の Ubuntu の基盤設定（locale=`ja_JP.UTF-8` / timezone=`Asia/Tokyo` / `/etc/wsl.conf`）。
- 言語ランタイム、コンテナ、Kubernetes ツール群、クラウド CLI、PostgreSQL の導入。
- opt-in のユーザー初期化（Git 設定・`~/.bashrc`・`~/.ssh` 権限）。
- 導入結果の検証（`playbooks/check.yml`、各 role の `validate.yml`、`script/check-system.sh`、Molecule）。

対象プラットフォームは WSL Ubuntu。`platform/ec2-ubuntu/` 等の他系統は本書の対象外（[AGENTS.md](../../AGENTS.md) 参照）。

## 3. 非対象（やらないこと / 人手に残すこと）

- **Windows 側の操作**: `wsl --update`、`wsl --set-default-version 2`、`C:\Users\<user>\.wslconfig` の作成は人手。Ansible は Windows 側を変更しない。
- **GitHub 画面操作**: SSH 公開鍵の GitHub への登録は手動。  
  ただし、登録済み秘密鍵の配布と `git clone` 時の利用は Doppler シークレット（例: `INTERNAL_GIT_SSH_PRIVATE_KEY`）で自動化可能。
- **`session-manager-plugin`** 等、明示的に「未対応 / 手動候補」とした項目。
- シークレットそのもののリポジトリ格納（`env/secret/` は gitignore、秘密は Doppler 駆動で注入）。

## 4. 初期セットアップ境界

最初のセットアップは、Windows 側と Ubuntu 側で責務を分ける。

- Windows 側で人手実施:
  - `wsl --update`
  - `wsl --version`
  - `wsl --set-default-version 2`
  - 必要に応じて `wsl --shutdown`
  - `.wslconfig` の作成
  - GitHub への SSH 公開鍵登録
- Ubuntu 側で人手実施:
  - `apt update/upgrade`
  - `build-essential`, `curl`, `git`, `zip`, `unzip`, `jq`, `htop`, `dnsutils`, `pkg-config`, `make` などの最低限パッケージ導入
  - リポジトリ clone
- Ubuntu 側で Ansible 実施:
  - `make setup-ansible`
  - `make syntax`
  - `make setup-base`
  - 必要に応じて `make setup-user`, `make setup`, `make setup-rust`

## 5. 設計原則

1. **layered 変数設計**: `group_vars/all.yml` をベースに、`group_vars/profiles/`、`group_vars/platform/`、`group_vars/domains/`、`env/config/setting.yaml`、`env/secret/credential.yaml` を責務別に重ねる。
2. **role 分割**: 関心ごとに role を分ける（`base` / `development-runtime` / `docker` / `kubernetes` / `aws` / `gcp` / `postgresql` / `user-bootstrap` / `cache`）。一覧は [03_実装カタログ.md](03_実装カタログ.md)。
3. **言語方針**: **ML は Python / 非 ML は Rust**。Node.js・Go はツールチェーン用途の基盤ランタイムとして常備。**Java / Kotlin は opt-in**（既定 OFF: `install_java_enabled` / `install_kotlin_enabled`）。
4. **クラウド方針**: メインは **GCP（gcloud）**。AWS CLI / Terraform / GitHub CLI は既存資産として温存する（撤去しない）。
5. **PATH は profile.d 集約**: Rust/Go/Java/Kotlin/gcloud などは `/etc/profile.d/*.sh` で PATH を通す。
6. **検証を組み込む**: 各 role は `validate_installation`（既定 true）で `validate.yml` を実行し、主入口の `playbooks/check.yml` と Molecule で回帰を確認する。
7. **入口は少なく、互換導線は分離する**: 主入口は `playbooks/main.yml` と `playbooks/check.yml` に寄せ、旧 `site*.yml` は `playbooks/compat/` で互換レイヤとして維持する。詳細は [03_実装カタログ.md](03_実装カタログ.md)。
8. **冪等性**: `creates:` ガードや `command -v` チェックで再実行に耐える。

## 5.1 実行モデル

- `playbooks/main.yml`
  - フル収束の主入口。`wsl_profile` と `execution_platform` に応じて profile/platform 上書きを読み込む。
- `playbooks/check.yml`
  - 実機状態確認の主入口。domain ごとの verify task を段階的に実行する。
- `playbooks/converge/`
  - `base` / `user` / `runtime` / `container` / `cloud` / `data` / `security` の中核導線。
- `playbooks/compat/`
  - 旧 `site*.yml` 系の互換導線。既存オペレーションを壊さずに新構造へ移行するために残す。

## 5.2 変数レイヤ

- `group_vars/all.yml`
  - 実行ベース、profile/platform 指定、共通パスの既定値。
- `group_vars/profiles/`
  - `full` / `minimal` / `runtime` / `ci` の導入範囲差分。
- `group_vars/platform/`
  - `wsl` / `molecule` の環境差分。timezone や service 制御の抑制もここで表現する。
- `group_vars/domains/`
  - `core` / `runtime` / `container` / `security` / `cloud` / `data` / `user` の domain 別設定。
- `env/config/setting.yaml`
  - 個人設定の正本。Git の `user.name` / `user.email` / `init.defaultBranch` はここを優先する。
- `env/secret/credential.yaml`
  - sudo password などのローカル秘密入力口。gitignore 対象。

## 6. ポート方針

WSL ネイティブ（apt）PostgreSQL と Docker コンテナの PostgreSQL はポートを分離して衝突を避ける。

| 用途 | ポート |
|---|---|
| WSL apt PostgreSQL | `5432` |
| Docker PostgreSQL dev | `5433` |
| Docker PostgreSQL test | `5434` |
| Docker PostgreSQL dwh | `5435` |

## 7. シークレット / 設定の置き場所

- `.env` — ローカル接続の代表値。
- `env/config/setting.yaml` — 非秘密の設定のみ。
- `env/secret/credential.yaml` — 秘密（gitignore 対象）。
- 本番/クラウド展開は Doppler 駆動（[bootstrap-from-doppler.sh](../../platform/wsl-ubuntu/script/bootstrap-from-doppler.sh)）でシークレットを注入する。詳細は [04_運用.md](04_運用.md)。
- `INTERNAL_GIT_SSH_PRIVATE_KEY` のような登録済み SSH 秘密鍵も Doppler から注入できる。GitHub 側の公開鍵登録自体は手動のまま残す。
