# WSL Ubuntu Ansible 基盤再設計計画

> **⚠️ 本書は再設計の背景計画です。現行仕様の正本ではありません。**
> 現行の仕様・運用・実装は [`docs/wsl-ubuntu-ansible/`](../../../../docs/wsl-ubuntu-ansible/) を正とし、本書は「今後長く使う前提でどのように作り直すか」を定義するための設計メモとして扱います。

## 1. 目的

この計画の目的は、いま動いている WSL Ubuntu Ansible を単に整理することではなく、次の条件に耐える **長期運用向けの土台** へ作り直すことです。

- role やツール追加のたびに playbook と変数参照が崩れない
- テスト導線を増やしても本体構成が濁らない
- `WSL 実機` と `Molecule 一時環境` の差分が構造として表現される
- 実行入口、設定、role、検証、docs の責務が混ざらない
- 今後 `cloud` や `security` を広げても、壊れ方が局所化される

## 1.1 今回の実装ゴール

この再設計計画は長期計画だが、今回のコーディングでの **明確な完了条件** は次に固定する。

- `playbooks/main.yml` と `playbooks/check.yml` を主入口として実装済みである
- 旧 `site*.yml` が `playbooks/compat/` に退避され、互換導線として残っている
- `group_vars/all.yml` の一枚依存が緩和され、少なくとも `profiles` / `platform` / `domains` の層が導入済みである
- `Makefile` と bootstrap 導線が新入口と variable layering を利用できる
- 基準テストとして `syntax` と `Molecule default/base/user-bootstrap` が通る

つまり、**Phase 1 と Phase 2 を実装完了にすること** を今回のゴールとする。

## 1.2 今日時点の完了見込み

2026-05-31 時点では、今回のゴールは **このターンで完了可能** な範囲にある。

- Phase 1: 実装完了
- Phase 2: 実装完了
- Phase 3 以降: 計画済み、未完了

## 2. 現状診断

現状は「動くが、継ぎ足しが効きすぎて全体設計の輪郭が曖昧」な状態です。

### 2.1 症状

- 入口 playbook が増え、差分が `構造差` ではなく `コピペ差` になっている
- `group_vars/all.yml`、`config.yml`、`env/config/setting.yaml`、`env/secret/credential.yaml`、Molecule 専用変数が並立し、設定責務が分散している
- role の責務が「パッケージ導入」「設定反映」「検証」「WSL 固有分岐」を同時に抱えやすい
- `playbook_dir` 相対参照や root 前提の探索に依存し、レイアウト変更に弱い
- `site-check.yml`、role 内 `validate.yml`、Molecule verify の境界が曖昧で、同じ検証責務が複数箇所に現れやすい
- docs 上は正本と背景資料を整理できたが、コード上の責務分離はまだ途中

### 2.2 根本原因

- 初期の「単一 playbook で素早く前進する構成」から、拡張後のアーキテクチャへ明示的に移行していない
- 変数設計が「どこに何を書くか」より「今使える場所に書く」で増築されてきた
- 実行導線の増加に対して、共通 orchestration 層がない
- テストを追加したが、`test strategy` が先に定義されていない

## 3. 再設計の基本方針

### 3.1 設計原則

- **入口は少なく、内部は明示的に分ける**
- **設定は層で分ける**
- **role は単機能・単責務に寄せる**
- **WSL 実機差分と Molecule 差分は変数層で表現する**
- **検証は階層化し、重複を意図的に減らす**
- **互換レイヤを挟みながら移行する**

### 3.2 非目標

今回の再設計では、次は主目的にしない。

- すべての role を一気に作り直すこと
- すべての tool version を再選定すること
- Windows 側設定や GitHub UI 操作を自動化すること
- いきなり完全な multi-host Ansible 基盤へ広げること

## 4. 目標アーキテクチャ

## 4.1 責務分離モデル

Ansible 本体を、次の 5 層で扱う。

1. **Execution Layer**
   - 何を、どのプロファイルで、どこまで実行するか
2. **Configuration Layer**
   - 既定値、環境差分、個人差分、秘密情報の上書き順
3. **Domain Layer**
   - base / runtime / container / cloud / data / user / security などの業務境界
4. **Verification Layer**
   - syntax / lint / smoke / verify / molecule / 実機 check
5. **Documentation Layer**
   - 正本 docs と背景資料の責務分離

## 4.2 将来の推奨ディレクトリ像

```text

├── playbooks/

│   ├── check.yml
│   ├── converge/
│   │   ├── base.yml
│   │   ├── runtime.yml
│   │   ├── user.yml
│   │   ├── cloud.yml
│   │   ├── data.yml
│   │   └── container.yml
│   └── compat/

│       └── site-selective.yml
```

### 4.3 この構造で解消したい問題

- playbook の増殖を `compat/` と `converge/` に分離し、役割を固定する
- `group_vars/all.yml` 一枚依存をやめ、`defaults` / `profiles` / `platform` へ分ける
- `development-runtime` のような複合 role を、必要に応じて domain role へ分解する
- 互換用 `site*.yml` を残しつつ、中核の実装は `main.yml` 系へ寄せる

## 5. 実行モデルの再設計

### 5.1 入口戦略

長期的には **主入口を 2 本** に絞る。

- `playbooks/main.yml`
  - 本番の収束入口
- `playbooks/check.yml`
  - 実機状態確認入口

それ以外の `site*.yml` は直ちに消さず、`playbooks/compat/` 配下の互換導線として残す。

### 5.2 profile 戦略

現在の `site-base.yml`、`site-rust.yml`、`site-selective.yml` が担っている差分は、最終的に `playbook の種類` ではなく **profile 変数** へ寄せる。

例:

- `profile=minimal`
- `profile=full`
- `profile=runtime`
- `profile=ci`

これにより、実行入口を増やさずに対象セットを切り替えられる。

### 5.3 tag 戦略

profile と role tag を併用し、次の粒度で再実行可能にする。

- `core`
- `runtime`
- `container`
- `cloud`
- `data`
- `security`
- `user`
- `verify`

## 6. 設定モデルの再設計

### 6.1 変数の責務分離

今後は次の優先順位で変数を解決する。

1. role defaults
2. inventory defaults
3. profile vars
4. platform vars
5. local env config
6. secret vars
7. extra-vars

### 6.2 現在のファイルの扱い

- `group_vars/all.yml`
  - 当面の SSOT として維持しつつ、将来的に `defaults.yml` と domain 別 vars へ分割
- `config.yml`
  - 「component on/off」の暫定 UI として維持し、最終的には profile へ吸収
- `env/config/setting.yaml`
  - 個人設定の正本として維持
- `env/secret/credential.yaml`
  - 秘密情報のローカル入力口として維持
- `molecule/resources/common-vars.yml`
  - 将来的に `platform/molecule.yml` 相当へ寄せる

### 6.3 導入したい規約

- version 変数と feature flag を分ける
- `*_enabled` と `install_*` の命名を統一する
- Molecule 用の抑制変数は `platform_*` または `molecule_*` に寄せる
- secret を必要とする role は、入力変数を role interface として明示する

## 7. role 境界の再設計

### 7.1 domain ベースへ寄せる

現状の role は大枠では悪くないが、長期的には次のように境界を強める。

- `core_base`
  - locale / timezone / apt / pipx / uv / doppler / 基本パッケージ
- `core_cache`
  - cache と download/workdir のみ
- `user_bootstrap`
  - 個人設定のみ
- `runtime_*`
  - 言語ごとに独立
- `container_*`
  - Docker と Kubernetes を分離
- `cloud_*`
  - AWS と GCP を分離
- `data_postgresql`
  - PostgreSQL と Neon CLI
- `security_tools`
  - セキュリティ CLI と hardening を分ける余地を残す

### 7.2 role 内構造の標準化

各 role で最低限そろえる構造:

```text
tasks/
  main.yml
  install.yml
  configure.yml
  verify.yml
defaults/
  main.yml
meta/
  main.yml
```

必要に応じて `handlers/` を追加するが、WSL 実機で systemd 依存が弱い role は handler 依存を増やしすぎない。

### 7.3 role interface の明文化

各 role について、将来的に次を README か docs へ持つ。

- 入力変数
- 外部依存
- 生成物
- verify 方法
- Molecule scenario 対応状況

## 8. テスト戦略の再設計

### 8.1 テスト階層

今後のテストは、次の 5 層に固定する。

1. `syntax`
2. `lint`
3. `smoke`
4. `molecule`
5. `host-check`

### 8.2 各層の責務

- `syntax`
  - YAML / playbook / role の構文破壊を防ぐ
- `lint`
  - 実装ルール逸脱を検知する
- `smoke`
  - 入口導線が最低限動くことを確認する
- `molecule`
  - role 単位の収束性 / 冪等性 / verify
- `host-check`
  - WSL 実機で導入結果が期待状態か確認する

### 8.3 現在との接続

- `make syntax` / `make lint` / `make check` / `make molecule-test*`
  - 当面維持
- `site-check.yml`
  - 長期的には `playbooks/check.yml` へ改名・統合
- Molecule
  - `default` / `base` / `user-bootstrap` を基準 scenario とし、そこから他 domain を積み上げる

## 9. ドキュメント戦略

### 9.1 正本の責務

引き続き、現行の正本は `docs/wsl-ubuntu-ansible/` に置く。

### 9.2 背景資料の責務

`ansible/docs/` は今後、次の 3 種類だけを置く。

- 再設計計画
- 採用前の比較検討
- 旧設計アーカイブ

### 9.3 禁止したい状態

- 仕様と現状実装の正本が `ansible/docs/` に戻ること
- 旧設計サンプルが実装参照先として再利用されること

## 11. 優先順位

今すぐ着手すべき順は次です。

1. 実行導線の再定義
2. 設定モデルの分離
3. role interface の固定
4. テスト責務の固定
5. role 分割と compat 整理

## 12. 破綻条件

この再設計が失敗する典型は次です。

- 新設計を入れる前に、現行運用導線を大量に消す
- vars 分離より先に role 分割へ着手する
- Molecule と実機差分をコード分岐で埋め続ける
- profile を作らず playbook を増やし続ける
- docs 上だけ整理し、Makefile や bootstrap が旧導線のまま残る

## 13. 成功条件

この計画が成功したと判断する条件は次です。

- 主入口が `main` と `check` に収束している
- profile で導入対象を切り替えられる
- variables の追加先が明確
- role ごとに interface と verify 方法が定義されている
- `default` / `base` / `user-bootstrap` / `runtime` の Molecule が安定 green
- docs の正本と背景資料の役割が再び混ざらない

## 14. 次アクション

この計画から次に切る実作業は、次の順がよい。

1. `base` と `user-bootstrap` の role interface を明文化する
2. `development-runtime` の role 分割方針を決める
3. `check.yml` を domain 別 verify task に分解する
4. `runtime` / `postgresql` / `docker` の Molecule を順番に green 固定する
5. compat 導線の縮退計画を作る
