# Hono Backend ― 実装状況サマリ

> **対象ユースケース**: PCデスクトップ向け管理画面・業務アプリケーション
> スマートフォン向けのレスポンシブデザインは考慮されていません。

---

## 技術スタック

| カテゴリ | 技術 |
|---------|------|
| Runtime | Node.js |
| Framework | Hono v4 |
| Database | MongoDB (mongodb v7) |
| Cache/Queue | Redis (Upstash @upstash/redis) - Optional |
| DI | Awilix v12 |
| Validation | Zod v4 |
| Logging | Pino v10 |
| Template | EJS |
| 認証 | Clerk (@hono/clerk-auth) |
| ファイルアップロード | Cloudinary v2 |
| Test | Vitest v4, Playwright |
| Secret管理 | Doppler |

### 読み込み優先順位

1. 環境変数 (Doppler, K8s Secret envFrom)
2. ConfigMapファイル (`env/config.json` or `CONFIG_FILE_PATH`)

Secret類は環境変数のみ対応（ファイル非対応）。

### env/config.json

開発環境向けのデフォルト設定ファイル。Secret以外の設定を格納。

```json
{
  "projectName": "starter",
  "appEnv": "local",
  "appMode": "origin",
  "port": 8001,
  "logLevel": "debug",
  "mongoDbName": "starter",
  "authEnabled": false,
  "alertEnabled": false,
  "rateLimitPerMinute": 1000
}
```

| キー | 型 | 説明 |
|------|-----|------|
| projectName | string | プロジェクト名 |
| appEnv | string | 環境 (local/development/staging/production) |
| appMode | string | 動作モード (origin/stub) |
| port | number | サーバーポート番号 |
| logLevel | string | ログレベル (debug/info/warn/error) |
| mongoDbName | string | MongoDBデータベース名 |
| authEnabled | boolean | Clerk認証の有効/無効 |
| alertEnabled | boolean | アラート機能の有効/無効 |
| rateLimitPerMinute | number | レート制限（リクエスト数/分） |

本番環境ではDopplerまたはK8s ConfigMapで管理。

## 認証機能 (AUTH FEATURE)

### 概要

Clerk による認証機能。`AUTH_ENABLED=true` で有効化。

### ロール階層

| Role | Level | 説明 |
|------|-------|------|
| owner | 100 | 全権限（システム管理含む） |
| admin | 75 | 管理者権限 |
| operator | 50 | 操作権限（設定変更可） |
| viewer | 25 | 閲覧のみ |

### ロール解決の優先順位

1. Clerk `sessionClaims.publicMetadata.role` (有効なロールの場合)
2. `ownerAdminUserIds` に含まれるユーザー → `owner`
3. デフォルト → `viewer`

### 認証ルート

| パス | 説明 |
|------|------|
| `/auth/login` | ログインページ（Clerk UI） |
| `/auth/status` | 認証ステータス API |
| `/auth/me` | 現在のユーザー情報 API |

### ミドルウェア

| 関数 | ファイル | 説明 |
|------|----------|------|
| `createAuthMiddleware` | `auth/clerk.ts` | Clerkセッション検証 |
| `createRequireAuthMiddleware` | `auth/clerk.ts` | 認証必須（未認証→ログインへリダイレクト） |
| `createAuthContextMiddleware` | `auth/clerk.ts` | AuthContext構築（ロール解決含む） |
| `createAuthorizeMiddleware` | `auth/clerk.ts` | ポリシーベースのアクセス制御 |
| `createAuthStrategy` | `auth/helpers.ts` | 認証ストラテジー生成 |
| `authRequired` | `auth/helpers.ts` | 認証必須ミドルウェアチェーン |
| `authOptional` | `auth/helpers.ts` | 認証任意ミドルウェアチェーン |

### アクセスポリシー (canAccess)

| ポリシー | 必要ロール |
|----------|-----------|
| `canAccess.dashboard` | owner のみ |
| `canAccess.settingsRead` | viewer+ |
| `canAccess.settingsWrite` | operator+ |
| `canAccess.adminPage` | admin+ |
| `canAccess.devtool` | admin+ |
| `canAccess.system` | owner のみ |
| `canAccess.jobsOperate` | operator+ |
| `canAccess.jobsView` | viewer+ |
| `canAccess.authenticated` | 認証済みユーザー全員 |

### 関連ファイル

- `src/domain/auth/types.ts` - ロール定義、AuthContext型
- `src/domain/auth/policies.ts` - ポリシー関数（hasMinRole, allOf, anyOf, not）
- `src/domain/auth/roleResolver.ts` - ロール解決ロジック
- `src/presentation/middleware/auth/clerk.ts` - 認証ミドルウェア
- `src/presentation/middleware/auth/helpers.ts` - authRequired(), authOptional()
- `src/presentation/middleware/auth/index.ts` - re-export
- `src/presentation/routes/auth/` - 認証ルート (page.ts, api.ts)
- `src/shared/authAuditLogger.ts` - 監査ログ

詳細は [01-clerk認証.md](../04-機能ガイド/01-clerk認証.md) 参照。

## Contracts層 (src/contracts/)

API契約の長期安定性（LTS: Long Term Support）を保証するための型定義層。

### 概要

| 項目 | 説明 |
|------|------|
| 目的 | APIレスポンス形式の後方互換性保証 |
| バージョン | lts-v1（破壊的変更時はv2を新設） |
| 凍結ルール | 公開後の型定義は変更不可 |

### ディレクトリ構成

```
contracts/lts-v1/
├── documents/           # 凍結された型定義
│   ├── micropost.ts     # LtsMicropostDocument, LtsMicropostCreateInput
│   ├── settings.ts      # LtsSettingsDocument, LtsSettingsInput
│   └── index.ts
├── types/               # 共通型定義
│   ├── ids.ts           # MicropostId, SettingsId, UserId
│   ├── time.ts          # LtsTimestamp, TimePolicy
│   ├── common.ts        # LtsDocumentBase, LtsMicropostStatus
│   └── index.ts
├── adapters/            # ドメイン ↔ 契約変換
│   ├── errors.ts        # AdaptError
│   ├── micropost.ts     # adaptMicropostToLtsV1, adaptLtsV1ToMicropost
│   ├── settings.ts      # adaptSettingsToLtsV1, adaptLtsV1ToSettings
│   └── index.ts
└── index.ts             # 公開API
```

### Documents（凍結型）

| Document | フィールド |
|----------|-----------|
| LtsMicropostDocument | id, title, body, creatorId, imagePath?, status, createdAt, updatedAt |
| LtsSettingsDocument | id, isCloud, isStub, aiModel, updatedAt |

### Types

| 型 | 説明 |
|-----|------|
| MicropostId | マイクロポストID（ObjectId形式） |
| SettingsId | 設定ID（timestamp-random形式） |
| UserId | ユーザーID（Clerk形式） |
| LtsTimestamp | Date型保証のタイムスタンプ |
| LtsMicropostStatus | 'draft' \| 'published' |

### Adapters

| 関数 | 説明 |
|------|------|
| `adaptMicropostToLtsV1` | Micropost → LtsMicropostDocument |
| `adaptLtsV1ToMicropost` | LtsMicropostDocument → Micropost |
| `adaptSettingsToLtsV1` | Settings → LtsSettingsDocument |
| `adaptLegacySettingsToLtsV1` | レガシー形式対応変換 |

### 使用方法

```typescript
import { adaptSettingsToLtsV1 } from '../contracts/lts-v1'

// コントローラでの使用例
const settings = await useCases.get()
const response = adaptSettingsToLtsV1(settings)
return c.json({ success: true, data: response })
```

## DI Container (src/di/)

### 設計方針

- **InjectionMode.CLASSIC**: 関数中心のため明示的 `resolve()` を採用
- **Factory不使用**: 全生成処理を `container.ts` に集約
- **ライフサイクル**: 非同期リソース（mongoClient）の破棄は呼び出し側で管理
