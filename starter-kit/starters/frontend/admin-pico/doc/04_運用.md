# 04. 運用

本ドキュメントは、`pico-admin-template` の日常運用手順をまとめる。

---

## 1. 前提

このリポジトリは静的 HTML / CSS / JS で構成される。

- バックエンド起動は不要
- ビルド済み CSS の生成は不要
- ローカル確認は静的配信だけでよい

---

## 2. 日常確認

### 2.1 ファイルを直接開く

単純な確認であれば `src/index.html` などをブラウザで直接開ける。

対象画面:

- `src/index.html`
- `src/pages/admin/analytics.html`
- `src/pages/admin/customers.html`
- `src/pages/admin/news.html`
- `src/pages/admin/ajax.html`
- `src/pages/admin/orders.html`
- `src/pages/admin/todo.html`
- `src/pages/admin/settings.html`
- `src/pages/auth/signin.html`
- `src/pages/auth/signup.html`
- `src/pages/auth/reset-password.html`
- `src/pages/errors/404.html`

### 2.2 ローカル静的配信

Playwright と同じ条件で確認したい場合は、簡易サーバを使う。

```bash
make serve
```

既定 URL:

```txt
http://127.0.0.1:4173/index.html
```

## 4. 変更時の確認ポイント

### 4.1 レイアウト

- デスクトップでサイドバーが常時表示されるか
- 本文が左に寄りすぎていないか
- KPI が横並び前提で崩れていないか
- サイドバー幅とナビ密度が他ページと揃っているか

### 4.2 配色

- 白いカード面が残っていないか
- 白いテーブル面が残っていないか
- ライムが通常情報に使われすぎていないか
- ピンクが通常情報に混入していないか
- チャートが大きすぎる、または読めないほど小さすぎる状態になっていないか

### 4.3 リンク

- 削除済みページへのリンクが残っていないか
- 12 画面だけで自己完結しているか

### 4.4 favicon

- 既定が `admin-dashboard.svg` になっているか
- 用途別 favicon の差し替えが反映されるか

---

## 5. 画面追加ルール

新規画面を増やす場合は以下を守る。

1. `Pico` 標準要素を優先する
2. `admin-*` クラスは必要最小限にする
3. デスクトップファーストで設計する
4. ダーク固定を崩さない
5. 追加後は `Playwright` の対象拡張を検討する

---

## 6. 禁止事項

- `Tailwind` の再導入
- `Alpine.js` の再導入
- `light / dark` 切替の追加
- 削除済みページの無秩序な復活
- 白背景 UI の混入

---

## 7. 障害対応

### 症状: ページが崩れる

確認順:

1. `src/css/admin-theme.css`
2. `src/css/admin-layout.css`
3. `src/css/admin-components.css`

### 症状: E2E が落ちる

確認順:

1. `cd src && node scripts/serve-static.mjs` で対象 HTML が配信できるか
2. `src/playwright.config.js` の `baseURL` と `webServer` が一致しているか
3. 画面内の見出しやボタン名を変更していないか

### 症状: 白背景が残る

確認順:

1. `admin-card`
2. `admin-table`
3. `input/select/textarea`
4. `thead/tr/th`
