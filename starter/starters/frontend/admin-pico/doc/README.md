# doc/ — ドキュメント運用ルール

本リポジトリのドキュメント群の役割、優先順位、更新規約を定義する。実装資産は `../src/` を正とし、記述がずれた場合はコード側に合わせて修正する。

## 1. ドキュメント一覧

| ファイル | 役割 | 更新契機 |
|---|---|---|
| [`01_仕様と設計.md`](./01_仕様と設計.md) | 現行仕様とデザイン原則の定義 | 要件変更時 |
| [`02_移行ロードマップ.md`](./02_移行ロードマップ.md) | 未処理タスク、残課題の整理 | 方針変更時 |
| [`03_実装カタログ.md`](./03_実装カタログ.md) | `src/` 配下の実装棚卸し | ファイル増減時 |
| [`04_運用.md`](./04_運用.md) | ローカル起動、E2E、確認手順 | コマンド変更時 |
| [`CHANGELOG.md`](./CHANGELOG.md) | 主な変更履歴 | リリース整理時 |
| [`../README.md`](../README.md) | 入口説明 | 構成変更時 |

## 2. 優先順位

矛盾した場合の優先順位は以下。

```txt
01_仕様と設計.md > 02_移行ロードマップ.md > README.md
```

`03_実装カタログ.md` と `04_運用.md` は上記を補足する従属文書として扱う。`02_移行ロードマップ.md` は未処理タスクだけを残す。

## 3. 更新ルール

- `src/` のファイル追加、削除、移動を行ったら `03_実装カタログ.md` を更新する。
- 起動方法、テスト方法、入口パスが変わったら `04_運用.md` と `README.md` を同時更新する。
- 画面方針や配色方針が変わったら `01_仕様と設計.md` を先に更新する。
- 大きな構成変更や未処理タスクが出たら `02_移行ロードマップ.md` に残す。

## 4. 書き方

- 日本語で書く。
- パスは現行構成に合わせて `src/...`、`doc/...` を明記する。
- コマンドは可能な限りルート `Makefile` から叩ける形を優先する。
- 実装とずれる推測記述は残さない。

## 5. 追加時の注意

1. まず既存文書へ追記できないか確認する。
2. 新規追加したらこの一覧と `README.md` のドキュメント節を更新する。
3. ファイル移動があった場合はリンク切れを必ず確認する。
