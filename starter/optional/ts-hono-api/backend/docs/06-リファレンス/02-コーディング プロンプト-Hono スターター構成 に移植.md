# 個人利用アプリ向け

## 既存実装 → Hono v4 スターターへの

## 思想抽出型・再構築手順書（汎用・提出版）

---

## この手順書の目的（固定）

この手順書は以下の状況を想定する：

* 既に動作している **旧アプリ（技術・構成は不問）** が存在する
* 新しい **Hono v4 + Layered Architecture スターター** を基盤にしたい
* **原則としてコードはコピーしない**
* しかし **機能・思想・UXは引き継ぎたい**

**ゴール**
**「旧アプリの“思想だけ”が、新スターター上で自然に再現されている状態」**

---

## ❗ 例外ルール（最重要）

> **既存アプリがすでにクラウドDBで稼働しており、
> そのデータ構造をそのまま引き継ぐ場合に限り、
> データモデル（Entity / 型定義）はコピーを許可・推奨する。**

### 理由

* 既存DBはすでに **運用実績のある Single Source of Truth**
* データモデルを再設計すると以下のリスクが発生する：

  * データ不整合
  * 移行コストの増大
  * 二重定義・仕様分岐

**思想抽出の対象は「振る舞い・責務・UX」であり、
データモデルそのものは抽出対象ではない。**

---

## 絶対原則（全アプリ共通）

1. **コードは資料ではない**

   * 参照対象は「挙動・意味・データ」
   * ただし **データモデル定義は例外**

2. **想像で設計しない**

   * 実データ・実画面・実操作を必ず確認する

3. **スターターの思想が最優先**

   * 旧実装よりスターター構成を正とする

4. **最初は必ず最小構成**

   * 後で足せるものは初回移植では削る

---

## Phase 0：移植前インフラ・前提確認（必須）

> ⚠️ このPhaseを省略した移植は高確率で失敗する

### 0-1. プロジェクト固有設定の所在確認

| 種類      | 設定場所              | 理由       |
| ------- | ----------------- | -------- |
| プロジェクト名 | `env/config.json` | プロジェクト固有 |
| DB名     | `env/config.json` | 他アプリと分離  |
| 接続文字列   | Secret Manager    | 環境共通     |
| APIキー   | Secret Manager    | 回転前提     |

**判断基準**
👉「複数プロジェクトで共有されるか否か」

---

### 0-2. 実データの確認（必須）

1. DBに接続
2. コレクション / テーブル一覧を取得
3. 各主要データの **実レコードを最低1件確認**

禁止事項：

* 想像でEntityを作る
* 旧コードのみを根拠に判断する

**原則**
**データ構造 = ドメインの真実**
**この真実に合わせて Entity / 型定義をコピー・作成する**

---

### 0-3. 認証・権限制御の確認

* 認証ON/OFFの切替有無
* 管理者 / 一般ユーザーの区別
* 認可単位（機能単位 / 画面単位）

※ この確認を後回しにすると Usecase 設計が破綻する

---

## Phase 1：思想抽出（実装禁止フェーズ）

> このPhaseでは **1行もコードを書かない**

### 1-1. ドメインの言語化

文章で以下を定義する：

* 何を管理するアプリか
* 管理対象の最小単位
* その単位が持つ状態
* ユーザーが実行できる行為

**名詞（Entity）と動詞（Usecase）を明確に分離する**

---

### 1-2. CRUDではなく「意味」で整理する

* 作成 / 更新 / 削除 ではなく
* 登録する / 公開する / 無効化する / 並べ替える

**CRUDは結果であり、目的ではない**

---

### 1-3. 画面とAPIを分離して考える

* 画面：体験
* API：手続き

画面を持たないAPI、
APIを直接叩かない画面、
どちらも許容される。

---

## Phase 2：Domain設計（最初に書くコード）

### 2-1. Entity設計

* フィールドは **実データに完全準拠**
* 状態・制約・不変条件を明示
* 便利メソッドは持たせない

#### Entity作成ルール（必須）

* 既存クラウドDBを引き継ぐ場合：

  * **旧実装の Entity / 型定義をコピーする**
  * フィールド構成・型・必須性は変更しない
* 修正が必要な場合：

  * **移植完了後フェーズに延期**
  * 本手順書の対象外とする

**Domain設計とは「形を考えること」ではなく
既存の形に意味を与えることである。**

---

### 2-2. Repository IF 設計

* 永続化方式は記述しない
* Usecaseが必要とする操作のみ定義する

---

## Phase 3：Usecase設計（主役）

* 1ユースケース = 1目的
* 1ファイル = 1ユースケース
* 例外はすべて明示的に扱う

  * Not Found
  * Forbidden
  * Invalid State

Usecaseは **業務手順書** である。

---

## Phase 4：Infra（最後に書く）

* Repository IF を満たすだけ
* DB都合を Domain / Usecase に漏らさない
* 既存DB構造への **Adapter** に徹する

※ InfraでDomainを歪める実装は禁止

---

## Phase 5：Presentation（薄く・分離）

### Validation

* Zod等で入力を確定
* Usecaseには確定データのみ渡す

### Controller

* Validation
* Usecase呼び出し
* Response整形のみ

判断・分岐は禁止。

---

## Phase 6：ルーティングとUI配置

| 種類    | 配置                 |
| ----- | ------------------ |
| メイン機能 | `/dashboard/*`     |
| サブ機能  | `/dashboard/xxx/*` |

URLはUX設計の一部である。

---

## Phase 7：削る判断（重要）

初回移植では以下を削除してよい：

* 高度な検索
* 履歴・監査
* 自動化・最適化
* 管理者向け詳細設定

**後で足せるものは、今やらない。**

---

## 最終チェックリスト（全アプリ共通）

* [ ] 実データを確認したか
* [ ] Entityを想像で作っていないか
* [ ] 既存データモデルをそのままコピーしているか
* [ ] Usecaseが主役になっているか
* [ ] Controllerが薄いか
* [ ] スターターの思想を壊していないか
* [ ] 旧コード由来の命名を引きずっていないか

## まとめ（思想の整理）

| 領域         | 扱い                |
| ---------- | ----------------- |
| データモデル     | **コピーする（既存DBが正）** |
| ドメイン概念     | 抽出・再定義            |
| ユースケース     | 再設計               |
| UI / UX    | 再構築               |
| Infra      | Adapterとして実装      |
| Controller | 配線係               |

**データは現実、設計は思想。**
この分離を守ることが、本手順書の核心である。
